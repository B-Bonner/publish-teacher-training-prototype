{{ title }}

Designs for the different states content can be in, for both courses and organisation information.

These designs represent a minimum viable workflow, they separate the save action from the publish action.

User needs

As a user I need to publish content so that applicants can see it.

As a user I need to save changes without immediately publishing them so that I can check or preview content, ask for a review, or make corrections, before applicants see that content.

As a user I need to know when there are unpublished changes so that I can publish them.

We expect to discover further needs in research that can be prioritised and addressed in later iterations, such as:

  • needing to see what has changed since the last published version
  • needing to see a history of all published changes
  • needing to revert a draft back to a published state

Validation

Validate against word count when saving.

Validate all required fields have been completed before publishing.

State

A course or the ‘About your organisation’ section can be in one of these states:

  • New: The fields are all empty
  • Draft: The fields have been edited and those changes have not been published
  • Published: The fields have been published and no subsequent edits have been made.

Design notes

  • Indicate when there are unpublished changes
  • Hide the publish action from content in published state
  • Once published, indicate the last publish date, ‘last’ differentiates from ‘first published’
  • Keep preview and publish actions in ‘new’ state for familiarisation
  • Include a view on website link to see what the live version looks like, must be distinguished from preview
  • Explain what preview is and why it’s useful – we’ve observed confusion about preview in research
